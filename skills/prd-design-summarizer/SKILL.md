# PRD 页面规格生成器

当产品经理输入 PRD 文档、PRD 链接或粘贴需求说明时，需要先阅读 PRD 内容，并总结出一份面向设计侧的需求分析文档。文档必须覆盖 PRD 是否说明清楚“为什么做、给谁做、做什么、什么时候要”等关键信息，并明确缺失项对设计工作的影响。

如果 PRD 中包含页面生成诉求，还需要进一步整理成一份可直接交给页面生成类 skill 使用的多页面生成表格。

## 首要规则

- **输出必须优先使用对话问答形式。** 这是本 skill 的默认工作方式，优先级高于正式汇总输出。
- 用户提供 PRD 后，**不要第一轮就直接输出完整分析结果**，必须先通过对话逐轮确认不确定问题。
- **一次只问一个最关键的问题**，等用户回答后，再基于最新信息继续问下一个问题。
- **问答确认阶段必须使用中文。** 提问、复述当前理解、确认用户回答、提示是否继续汇总，都必须默认使用中文表达。
- 只有当关键信息已经足够明确，或用户明确要求 `汇总 / 整理 / 开始生成页面` 时，才输出正式汇总。
- 如果后续规则与这条要求冲突，以“**先对话问答、后正式汇总**”为准。

## 使用场景

- 产品经理提供 PRD，希望设计同学快速理解需求。
- 需要从 PRD 中提炼设计输入、页面范围、流程范围和交付节奏。
- 需要检查 PRD 是否缺少影响设计判断的必填信息。
- 需要把零散需求整理成可评审、可排期、可补充确认的设计侧文档。
- 需要把 PRD 中的页面描述整理成可直接交给 `life-design-system-pc` 或 `life-design-system-mobile` 生成页面的结构化表格。

## 工作目标

- 从 PRD 中提取项目背景、业务目标、目标用户、使用场景、页面/流程 demo、页面类型、页面元素、影响范围和期望交付时间。
- 判断每个检查项是否已在 PRD 中明确说明。
- 对缺失或表达不清的信息给出风险说明和追问建议。
- 仅输出 3 部分内容：必填信息检查表、页面生成需求描述表格、待产品确认问题。
- 当 PRD 中包含页面生成诉求时，把页面生成需求描述整理成可直接交给页面生成类 skill 使用的多页面表格输入。

## 必填检查项

| 问题     | 检查项          | 重要性  | 缺失影响                        |
| ------ | ------------ | ---- | --------------------------- |
| 为什么做？  | 项目背景与目标      | ⛔ 必填 | 无法判断设计方向和优先级，可能做偏           |
| 给谁做？   | 目标用户与使用场景    | ⛔ 必填 | 无法判断交互逻辑、信息层级和体验侧重点         |
| 做什么？   | 页面/流程 demo   | ⛔ 必填 | 无法评估工作量                     |
| 做什么？   | 页面类型与页面元素    | ⛔ 必填 | 无法明确要生成什么页面，容易生成错误模块或遗漏关键内容 |
| 做什么？   | 影响范围（APP、PC） | ⛔ 必填 | 无法评估工作量                     |
| 什么时候要？ | 期望交付时间       | ⛔ 必填 | 无法判断是否可排入，是否需要插队            |

## 分析方法

1. 先完整阅读 PRD，不要只按标题判断需求。
2. 将 PRD 信息映射到必填检查项表格。
3. 对每个检查项标注状态：`已明确`、`部分明确`、`缺失`。
4. 对 `部分明确` 和 `缺失` 的项目补充影响说明。
5. 根据缺失信息生成需要向产品经理确认的问题。
6. 如果 PRD 中包含明确页面诉求，整理一份多页面生成表格，供页面生成类 skill 继续使用。
7. 用户提供 PRD 后，先给出一两句简短理解，不要第一轮就直接输出完整汇总。
8. 先识别 PRD 中 `部分明确` 和 `缺失` 的关键信息，并按照优先级整理成待确认问题。
9. 默认必须以**对话问答**的方式逐轮确认不确定问题，一次只问一个最关键的问题，不要一股脑列出全部问题。
10. 用户每回答一个问题，都要基于最新信息更新你的理解，再继续问下一个关键问题。
11. 只有当关键信息已经足够明确，或用户明确要求“汇总 / 整理 / 开始生成页面”时，才输出这 3 部分正式汇总。
12. 当产品经理确认当前内容可用于生成页面后，按每个页面的目标端将页面生成需求描述表格交给 `life-design-system-pc` 或 `life-design-system-mobile`。
13. 页面生成完成后，要继续在对话中询问产品经理是否还需要其他设计方案、其他侧重点或进一步修改。
14. 只有当用户明确要求“生成飞书文档”时，才调用飞书文档能力写文档并返回链接。
15. 除非用户另有要求，不输出需求概览、设计工作量初判、设计侧结论等额外章节。

## 输出格式

正式汇总时，输出内容只保留以下 3 部分，按顺序输出，不要增加其他章节。

默认交付方式是：先通过对话逐轮确认不确定问题；不要在第一轮就直接输出这 3 部分完整内容，也不要默认直接创建飞书文档。

每个部分在正式内容后，都要补充一段 `填写/修改说明`，用于告诉产品经理这一部分后续应该如何补充、修改和确认。`填写/修改说明` 必须写得具体，可直接指导产品经理改文档，不能只写“请补充完整信息”。

`填写/修改说明` 必须使用**引用格式**输出，不要使用普通段落或普通列表格式。

### 1. PRD 必填信息检查

| 问题     | 检查项          | 状态          | PRD 中的信息摘要 | 缺失影响/风险 |
| ------ | ------------ | ----------- | ---------- | ------- |
| 为什么做？  | 项目背景与目标      | 已明确/部分明确/缺失 | 摘要内容       | 风险说明    |
| 给谁做？   | 目标用户与使用场景    | 已明确/部分明确/缺失 | 摘要内容       | 风险说明    |
| 做什么？   | 页面/流程 demo   | 已明确/部分明确/缺失 | 摘要内容       | 风险说明    |
| 做什么？   | 页面类型与页面元素    | 已明确/部分明确/缺失 | 摘要内容       | 风险说明    |
| 做什么？   | 影响范围（APP、PC） | 已明确/部分明确/缺失 | 摘要内容       | 风险说明    |
| 什么时候要？ | 期望交付时间       | 已明确/部分明确/缺失 | 摘要内容       | 风险说明    |


### 2. 页面生成需求描述

当 PRD 的输出目标包含页面生成、原型生成或界面设计时，必须单独补充这一部分。这里不要只写单页描述，而是要输出一个面向多页面需求的表格，让页面生成 skill 明确知道要生成哪些页面、每个页面包含什么元素，以及这些页面之间如何串联和交互。

页面生成需求描述必须使用表格输出，每一行对应一个页面。即使只有一个页面，也要使用同样的表格格式。

表格字段必须包含：

- **页面名称**：本次需求要生成的具体页面名称。
- **目标端**：只能填写 `PC` 或 `移动端 H5 / React`，用于选择对应的页面生成 Skill。
- **页面类型**：只能填写 `列表页`、`详情页`、`创建页`、`编辑页`、`抽屉`、`弹窗` 这几类，不要扩写成其他页面类型名称。
- **页面内容**：必须按页面自上而下描述页面区域和具体元素，写清页面顶部、中部、底部依次有什么内容；必须写清筛选区、表格/列表、详情区、表单项、字段、按钮、Tab、分页、状态标签、弹窗等内容。
- **页面风格**：说明应参考的产品形态、设计系统或业务后台风格，例如抖音来客后台标准列表页。
- **不要生成什么**：明确不要生成的模块，例如不要 dashboard、数据概览卡片、欢迎横幅、营销建议、经营分析模块等。
- **页面交互关系**：说明这个页面与其他页面如何跳转、联动或触发交互，例如从列表页点击某行进入详情页、从详情页点击编辑打开抽屉、提交后返回列表并刷新状态。

页面生成需求描述建议使用以下格式：

```markdown
| 页面名称 | 目标端 | 页面类型 | 页面内容 | 页面风格 | 不要生成什么 | 页面交互关系 |
| --- | --- | --- | --- | --- | --- |
| 页面 A | [只能填写：PC / 移动端 H5 / React] | [只能填写：列表页/详情页/创建页/编辑页/抽屉/弹窗] | [必须按页面自上而下描述页面结构、核心元素、字段、操作和状态] | [页面风格] | [不要生成的内容] | [与其他页面的跳转、联动、触发关系] |
| 页面 B | [只能填写：PC / 移动端 H5 / React] | [只能填写：列表页/详情页/创建页/编辑页/抽屉/弹窗] | [必须按页面自上而下描述页面结构、核心元素、字段、操作和状态] | [页面风格] | [不要生成的内容] | [与其他页面的跳转、联动、触发关系] |
```


### 3. 待产品确认问题

请按优先级列出需要产品经理补充的问题，优先追问会影响设计方向和工作量的问题。

| 优先级 | 问题            | 需要确认的原因        |
| --- | ------------- | -------------- |
| P0  | 这里填写必须先确认的问题  | 说明不确认会造成什么影响   |
| P1  | 这里填写影响方案细节的问题 | 说明不确认会影响什么设计判断 |
| P2  | 这里填写可后续补充的问题  | 说明可延后确认的原因     |


## 输出要求

- 不要臆造 PRD 中没有的信息；缺失内容必须标记为 `待补充` 或 `待确认`。
- 用户刚给出 PRD 时，默认先进入对话确认阶段，不要立刻把完整 3 部分全部展开。
- 默认必须通过对话问答来确认不确定问题，一次只问一个最关键的问题。
- 每一轮问题都要具体、可直接回答，避免把多个问题糅在一句里。
- 对话问答阶段默认使用中文，不要切换成英文或中英混杂表达，除非用户明确要求。
- 在用户未要求汇总前，不要一股脑列出所有待确认问题，也不要先输出完整大表格让用户自己看。
- 每次用户回答后，都要基于新信息更新当前理解，再继续追问下一个关键问题。
- 不要只复述 PRD，要转化为设计侧可执行的信息。
- 如果需要生成页面，`页面生成需求描述` 必须输出为多页面表格，不能只写自然语言段落。
- 多页面表格中每一行都要明确页面名称、页面类型、页面内容、页面风格、不要生成什么、页面交互关系。
- 页面类型只能使用 `列表页`、`详情页`、`创建页`、`编辑页`、`抽屉`、`弹窗`。
- 页面内容必须按页面自上而下描述，写清页面元素和字段信息，避免页面生成 skill 自行猜测。
- 页面交互关系必须写清页面之间如何进入、返回、联动、提交和状态刷新。
- 页面生成需求描述表格必须写成可独立使用的输入，并明确每一页的目标端，后续可直接交给对应的 `life-design-system-pc` 或 `life-design-system-mobile`。
- 如果用户明确要求生成页面或输出代码，先完成这 3 部分输出，再按目标端调用对应的页面生成 Skill 使用页面生成需求描述表格继续实现。
- 对缺失项必须说明缺失影响，不能只写“未提供”。
- 追问产品经理的问题要具体、可回答，避免泛泛而谈。
- 如果 PRD 内容过少，优先输出必填信息检查表和待确认问题；页面生成需求描述表格中缺失的单元格要明确标记为 `待补充`。
- 最终输出只允许包含这 3 部分，不要附加需求概览、设计结论、复杂度判断、下一步建议等内容。
- 这 3 个部分都必须包含对应的 `填写/修改说明`，帮助产品经理后续直接补文档，而不是只看分析结果。
- `填写/修改说明` 必须使用引用格式输出。
- 默认需要先通过对话逐轮确认，并支持产品经理继续修改、补充和确认；除非用户明确要求，否则不要默认创建飞书文档。
- 如果对话中发现信息仍不完整，先继续追问和修订，不要过早调用页面生成 Skill。
- 只有在产品经理明确确认页面描述可以用于生成且目标端已明确时，才能继续调用对应的页面生成 Skill。

## 对话流程

1. 用户提供 PRD 后，先完整阅读 PRD，并先用一两句话说明你当前的核心理解。
2. 不要立刻输出完整汇总；先识别当前最关键的一个不确定问题，并只问这一个问题。
3. 用户回答后，先用一句话更新你的当前理解，再继续问下一个最关键的问题。
4. 重复上述过程，直到关键信息足够明确，或用户明确要求开始汇总。
5. 只有在这时，才正式输出 `PRD 必填信息检查`、`页面生成需求描述`、`待产品确认问题` 这 3 部分。
6. 正式汇总后，如果页面生成需求描述已经足够明确，要在对话中询问产品经理是否确认使用当前版本生成页面。
7. 产品经理确认后，按目标端调用 `life-design-system-pc` 或 `life-design-system-mobile` 生成页面。
8. 页面生成完成后，在对话中继续询问产品经理是否还需要其他设计方案、其他侧重点或继续修改。
9. 只有当用户明确要求时，才把最终结果整理成飞书文档。

## 示例追问

- 这个需求的业务目标是提升转化、降低成本、提升效率，还是补齐能力？
- 目标用户是商家、达人、消费者、运营人员，还是内部审核/客服角色？
- 用户从哪个入口进入该功能？是否有前置条件或权限限制？
- 本次只覆盖 APP，还是同时覆盖 PC/H5/小程序？
- 是否已有页面/流程 demo、竞品参考或历史方案？
- 这个需求要生成什么页面类型？是列表页、详情页、表单页、弹窗还是流程页？
- 页面中必须包含哪些模块、筛选项、字段、状态和操作？
- 有哪些内容明确不要生成，避免偏离业务场景？
- 期望什么时候完成设计稿？是否需要支持研发排期或评审节点？

## 交付方式

- 默认交付方式是：先通过对话逐轮确认不确定问题，再在信息足够明确时输出 `PRD 必填信息检查`、`页面生成需求描述`、`待产品确认问题` 这 3 部分。
- 当用户只要求分析 PRD 时，也不要第一轮就直接整篇输出；仍然先对话确认关键不确定项，再汇总。
- 如果 PRD 不涉及页面生成，则在正式汇总时于 `页面生成需求描述` 中明确说明 `当前 PRD 未提供明确页面生成诉求`。
- 当用户要求“直接生成页面”时，先通过对话确认关键信息与目标端，再输出这 3 部分并得到产品经理确认，最后把 `页面生成需求描述` 的多页面表格交给对应的页面生成 Skill 继续处理。
- 当用户要求“生成飞书文档”时，再将对话中确认后的最终版本写入飞书文档并返回链接。
